# 翻译为中英对照

## 优先级

🥇 **默认：用当前运行的大模型（Agent 自己）逐块翻译**——无外部依赖、质量可控、术语可全局统一。长文按 3000–5000 字符分块逐块译。

🥈 **备选（仅用户明确要求且已配 `GEMINI_API_KEY`）**：

```bash
python3 scripts/translate_gemini.py <清洗后.md> <中英对照.md>
```

> ⚠️ Gemini 脚本若某分块失败会**静默保留英文原文**，译后必须搜一遍是否有整段未翻译的英文，发现就补译。

## 格式规则（务必遵守）

1. **标题单行双语**：`## English Title / 中文标题`（同一行 ` / ` 分隔）。
   **禁止**把中文标题另起一行成独立标题——否则一个章节在目录里占两行。
   文档自身的 Contents/目录清单也适用同一规则：每个目录项必须是
   单行配对。源文有类别时用 `**Category / 双语类别**: English Title / 中文标题 · p.NN`；源文无类别时直接用 `**English Title / 中文标题** · p.NN`。类别标签必须在源文结构中真实存在，禁止为对齐补造；逐项核对原始 `source.md`、清洗稿与最终双语稿，最终英文侧必须完整包含原始目录标题。
2. **正文排版按句末标点判断**：
   - 去掉行尾 Markdown 标记及闭合引号/括号后，只要英文原文或中文译文以明确句末标点
     （英文 `. ! ?`，中文 `。！？`）结束，就采用英文一段、中文一段，中间空一行。
   - 若英文和中文两侧都没有明确句末标点，则视为标签、短语、指标或非完整句，
     写成同一行 `English / 中文`，使版面紧凑。不要仅凭字符长度判断。
   - 标题、目录、表格、列表、引用块和代码仍优先遵循各自的结构规则。
   - **列表逐项单行双语**：每个列表项都写成
     `- English item / 中文条目`，无论列表项是否以句号结尾。禁止先列一组英文，
     再列一组中文；校验脚本会直接拒绝这种结构。
3. **保留所有结构**：`![](...)` 图片引用、HTML/markdown 表格、`> [!stat]` / `> [!definition]` 标注、列表、链接，原样保留。
4. **不加**分隔线、不加国旗 emoji。
5. **术语全局统一**：先扫全文建一份术语表（英文 → 选定中文译法），整篇沿用同一译法，避免同一个词忽中忽英、忽此译忽彼译。专业术语可「中文（English）」首次出现时标注，后文统一。
6. **无句末标点内容单行双语**：英文和中文均无明确句末标点的正文单元保持原有逐行粒度，写成 `English / 中文`。这包括短标签、指标名、短数值说明和其他非完整句；不要把多条内容合成长段，也不要先堆一组英文、再堆一组中文。
7. **署名引语**：仅当文本是明确的完整引语，且紧邻人名/职位署名时，使用标准 Markdown blockquote。英文、中文和署名保持在同一个连续 `>` 块中；署名可加粗，不改写原文。不要默认使用 `[!quote]`，公共上传器未证明原生支持时它会成为字面文本。
8. **近似符号**：正文中表示“约”的裸 `~` 改为 `≈`，避免两个波浪号被 Markdown 识别为删除线。代码块、行内代码和 URL 保持原样。

## 双语对照示例

```markdown
## Introduction / 引言

Standard RAG pipelines follow a static retrieve-then-generate paradigm.

标准 RAG 流水线遵循静态的「先检索再生成」范式。
```

## 语言判定

所有输入模式统一读不可变基准 `source.md` 的前 500 个正文字符（忽略 Markdown
标记、图片和 HTML 注释），统计中文字符占比：
- ≥30% → 视为中文内容，**跳过翻译**。
- <30% → 非中文（如英文论文/报告），**翻译为中英对照**。
