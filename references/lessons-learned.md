## 🔴 P0 — 反复犯过的严重错误（必须彻底杜绝）

### M01:【元规则】同类问题被反复指出 → 自动触发反思 + 主动全文同类排查
- **背景**：实战中用户先后指出"图漏了""统计框该高亮""样式化表格没保真""卡片被拍平+`$^{19}$` 乱码"——本质都是**同一类问题（PDF 富样式元素被拍平/漏处理）的不同实例**。每次只修被截图那一个，导致"修一个、冒一个"。用户明确要求："**当我多次明确指出同一个问题的时候，要自动触发反思机制，更新 skill。**"
- **根因**：①被动逐点修复，从不假设"还有同类实例"；②**有规则≠已落实**——多列卡片转表格的规则早已存在，却没据此全文扫一遍，于是半成品/漏网仍在；③把"更新 skill"当额外指令，而非复发时的默认动作。
- **正确做法（强制）**：
  1. 用户**第 2 次**（含）指出**同一类**问题（哪怕实例不同）→ 立即判定"系统性问题"，**不准**只修被指那一个。
  2. 把问题抽象成可检索特征（grep/脚本），**全文扫描列出所有实例**，**一次性全修**，再做"实例总数对账"。注意**"半成品"也算同类**（如多列卡片只转了一行表格、其余仍是散文本）。
  3. **自动反思**：同一轮里主动把根因+正确做法写进本文件，更新 SKILL 自检清单——不等用户催。
  4. **回报**：说明"这是 X 类问题，已全文排查到 N 处并全部修复"，而非只说"这一处修好了"。

### M02: 图表完整性核对不彻底——漏「一页多图」与「页码空档藏图」
- **背景**：抽查式"彻查"反复漏图（曾被用户连续 3 次指出），最终逐页渲染全本才一次揪出剩余漏图。
- **根因**：①一页多图只取第一张，第二张图注被当成 `##` 标题、后面没图；②按已知页码反推跳过空档页；③图注被解析成孤立标题，肉眼扫 markdown 不易察觉；④基于过期数据统计"图片数"对上了就以为没问题。
- **正确做法**：逐页低 DPI 渲染全本 → 颜色特征识别图表页 → 纵向像素聚类检测多图页（≥2 簇，命名加 `b`/`c` 后缀）→ **硬验收 `PDF 图表数 == 本地 ![ 引用数`**，不等就逐页定位。详见 [extraction.md](extraction.md) 第五节。

## 🟡 P1 — 富元素保真（识别 + 可移植表达）

### M03: 原图是「样式化表格/卡片」→ 保留原图 + 重建表格
样式化对照表/卡片表会被拍平成几十个散段落。**两件事都做**：①裁整张表格图插到标题下；②内容写成中英同格的 markdown/HTML 表（合并单元格用 `rowspan`/`colspan`）。详见 [rich-elements.md](rich-elements.md) 第四节。

### M04: 统计高亮框 / 定义卡片 → callout 标注（数字只出现一次）
左数字+右说明的统计框、`DEFINITION`+术语的定义卡片，常被拍平成 3–4 个孤立段落、术语被误标成标题。用 `> [!stat]` / `> [!definition]` 标注重建，数字/术语只出现一次。详见 [rich-elements.md](rich-elements.md) 一、二节。

### M05: 多列信息图 → 1 行 N 列 HTML 表格（不加标题）
并排 N 列卡片被解析成纵向堆叠 + 每列设 `###` 标题。重建为 1 行 N 列 HTML `<table>`，`<strong>` 加粗，**不设 `#` 标题**。复杂多列用表格而非多栏布局（更稳、透传）。

### M06: 脚注上标 `$^{N}$` → Unicode 上标
PDF 解析把脚注上标输出成 `$^{19}$` 这类 LaTeX 乱码。全文正则替换为 Unicode 上标（紧贴前字符），交付前必搜 `$^{` / `$_{` 归零。详见 [rich-elements.md](rich-elements.md) 第五节。

### M07: 标题层级 + 目录 + 单行双语
解析的 `#` 级别不可信（作者名/纯百分比/重复子标签都被标成标题），目录膨胀。按文档真实逻辑重排层级；冗余项降为加粗正文；章节序号 kicker 并入 H1；目录页重建为「一章一行」；标题单行双语 `## EN / 中文`（禁止中文另起一行）。详见 [rich-elements.md](rich-elements.md) 六、七节。

### M15: 目录类别一致不等于虚构章节；父子标题不能同级
- **背景**：目录前三项是正式 `Section 01–03`，第四项实际是领导团队与贡献者名单，却缺少同位置类别标签；正文中 `Parallel Shifts` 与其直接论述标题又被解析成同级 H3。
- **根因**：把“视觉一致”误解成统一套用 Section 编号；标题清洗只看单个标题样式，没有按章节语义建立父子层级。
- **正确做法**：目录统一使用「`双语类别: 双语标题 · 页码`」，但类别必须忠于源文；非正式章节使用 `Appendix / 附录`、`Directory / 名录` 等准确标签，禁止虚构 `Section 04`。标题按 H1 大章节 → H2 章节内主题 → H3 主题下论述 → 必要时 H4 命名子项重排；全文检查相邻同级标题以及主题标题后的同级论述标题，并做最小修复。

### M16: 目录格式化不能截断标题；短指标不能按普通段落翻译
- **背景**：原始目录完整保留了标题，但清洗重建时将其缩短；后续翻译和类别修正都沿用了截断值。另一组无句号的标签与数值说明被装配成“整组英文 + 整组中文”，破坏了逐项对应。
- **根因**：目录清洗只按最终视觉格式手工重写，没有做 `source.md → source_clean.md → final.md` 的标题包含关系对账；翻译装配只区分标题与正文，未单独识别短标签/指标行。
- **正确做法**：目录类别和标题分字段处理，类别可规范化，标题英文必须完整保留并自动做包含关系校验；正文大章节内部的解析 H1 统一复核并降为 H2。无句号结尾的短标签、指标名和短数值说明逐行写成 `English / 中文`，完整句子仍采用英文段后接中文段。

### M17: 视觉对齐不能补造目录类别；裸波浪号会变删除线
- **背景**：无类别的目录项被人为补上 `Directory / 名录`；统计指标中的两个裸 `~` 被 Markdown 上传链路解析成删除线；署名引语仍散落为普通段落。
- **根因**：把格式一致误当成语义一致；未区分“约”的文本符号与 Markdown 删除线语法；富元素识别遗漏了“引语 + 署名”这一稳定结构。
- **正确做法**：目录类别必须能由源文独立结构标签佐证，无类别就只显示完整标题。代码/URL 外表示近似值的 `~` 统一转为 `≈`，校验器拒绝近似值裸波浪号和成对裸波浪号。全文识别明确署名引语并统一为标准 `>` 引用块；不默认使用未验证的 `[!quote]`。

### M08: 表格翻译中英同格，别整行/整表另起
翻译时把中文整行/整表另起会让目录/正文极乱。文本单元格内 `English<br>中文` 同格双语，数据/百分比格不必翻译。

### M18: 双语列表必须逐项同一行，不能按语言分成两组（2026-07-15，用户反馈）
- **问题**：英文列表五项后又单独出现中文列表五项，读者无法快速逐项对应，版面长度也翻倍。
- **根因**：翻译装配只对标题和无句号短句执行单行双语，错误地让“列表结构优先”覆盖了
  双语逐项配对；完整句列表因此被按英文组、中文组堆叠。
- **正确做法**：列表不受句末标点规则影响，每一项固定写成
  `- English item / 中文条目`；保留原列表顺序、编号和嵌套层级，禁止先列全部英文再列全部中文。
- **脚本固化**：`bilingual_validation_core.py` 的
  `validate_bilingual_list_items_paired` 检查每个自然语言列表项同时包含英文、中文和
  ` / ` 分隔；发现分组堆叠直接校验失败。

### M19: 图片解析元数据不得进入正文；普通金额不转义美元符号（2026-07-15，用户反馈）
- **问题**：`[IMAGE]...[/IMAGE]` 或图片后的 `<image_ocr>` 被当成正文，最终页面在图片后
  又附上 OCR 表格、原始 OCR 标签和中文复述；普通金额被写成 `\$`，影响阅读。
- **根因**：将提取阶段的图片定位/OCR 元数据误当成需要满足“原文零删改”的正文；同时
  为规避 Markdown 公式解析而无差别转义 `$`。
- **正确做法**：图片块及其 OCR 只用于下载和核对图片，最终仅保留本地图片引用；不得
  输出图片标题、描述、OCR 原文、OCR 翻译或由 OCR 衍生的表格，除非它们在图片块外本来
  就是独立正文。普通货币金额直接使用 `$`；仅真实公式交给公式模式处理。
- **脚本固化**：`lexiang_fetch.py` 在 Step 1 即处理行内 `![](/assets/...)<image_ocr>`
  引用、下载图片并剔除全部 OCR 元数据；校验器忽略源文件中的图片 OCR 元数据，并拒绝
  最终文件残留 `[IMAGE]`、`<image_ocr>`、图片元数据标签或 `\$`。

## 🟢 P2 — 提取技巧（备忘）

### M09: 文字与图片分两条路 + clip 必传
`get_text()` 只抽文字（矢量图/表格结构会丢，预期）；图形用 pymupdf 渲染裁剪。**绝不能** `get_pixmap()` 不传 `clip`（会截整页含正文）。报告页宽约 1120pt，clip 右边界到 ~1060，别只裁到中线。

### M10: 卡片检测裁剪 + 目视核对
报告图表由几十个碎片拼成，用「最大填充矩形=卡片背景」检测裁剪整张；但对纯文字卡片也会命中，**必须 Read 预览核对**只留真数据图表。装饰元素（封面/头像/箭头/孤立数字块）一律剔除。

### M11: 乐享文档链接 → AI 解析 + lxapi assets 下图片（默认路径）
- **背景**：乐享内 PDF/文档可通过 `entry_describe_ai_parse_content` 拿到结构化 markdown（`data.content`），含 `[IMAGE]` 块与 `/assets/{asset_id}`。
- **图片下载（两步）**：
  1. `GET https://lxapi.lexiangla.com/cgi-bin/v1/assets/{asset_id}` + Bearer `lxmcp_` → 返回 JSON `{"url":"...", "mime_type":"image/png"}`（**不是**直接返回二进制）
  2. GET 签名 `url` 下载图片（无需额外鉴权）
  **不是**直接 GET 页面相对路径 `/assets/xxx`（那会 401）。
- **填充位置**：按 `[IMAGE]` 块在原文中的顺序替换为 `![标题](images/img_NNN.ext)`，产出 `source.md`。
- **降级**：assets 失败时 `--download-pdf` + pymupdf 补裁（见 [extraction.md](extraction.md)）。
- **装饰图**：解析器仍会把封面/头像/箭头标成图片，Step 5 必须剔除，只留真数据图表。
- **脚本**：`scripts/lexiang_fetch.py`（仅获取，与 upload 无代码依赖）。

### M12: 上传乐享必须复用独立 `upload-markdown-to-lexiang` Skill
- **背景**：若 Agent 逐段调 MCP（`entry_import_content_to_entry` + 每张图 `block_apply...`），每段/每张图消耗一轮 tool call token，长文极贵。
- **正确做法**：翻译完成后定位公共 Skill，检查 `cli_api=1`，再运行其
  `scripts/lexiang_upload.py upload ...`。本 Skill 不保存上传代码副本。
- **禁止**：上传阶段 Agent 手动调 MCP 工具，或把公共脚本复制回本 Skill。

### M13: 不能把“协议声称支持公式”当成“OpenAPI 实测可写”
- **背景**：MCP resource 声称 block-markdown 支持 `$`formula`$`，但 2026-07-13 的端到端实测中，`block_update_page` 会将其转成普通 `text_run`，`block_fetch_page` 返回转义后的 `\\$\\`...`\\$`；`<Equation>` MDX 和 `equation` block_type 也会被当前 OpenAPI 拒绝。旧代码只检查 LaTeX 字符还在，因此把“普通文本未丢”误判为“公式块成功”。
- **正确做法**：默认 `--formula-mode unicode`，行内与独立公式统一转为可编辑、可搜索的 Unicode 文本（如 `Kᵥ/ρ`、`B₄C`）；只有用户明确要求时才用 `--formula-mode image` 将表格外公式转图。`hybrid` 仅作为 `unicode` 的兼容别名。`--formula-mode native` 仅用于未来 API 修复后的探测，线上对账必须确认未转义的 `$`...`$` 数量完全一致，否则失败。
- **自检**：unicode/image 模式验证所有公式信息均已物化；image 模式额外对账线上 image block 总数；native 模式必须验证真实公式标记，不能只搜公式正文。

### M14: 分段边写边改会留下半成品，且图片后不能再用 update_page 重写
- **背景**：旧脚本先覆盖文本、再传图、再用 `block_update_page(update_content)` 追加后文；当前 reconcile 在图片后插入文本会报 `unsupported block operations`。中途失败时线上页面已经被破坏。
- **正确做法**：本地先构建 `UploadPlan`，一次性校验标题、公式、图片文件和段落锚点；首段用 `replace_content`，图片用上传会话 + image block，后续文本先 `block_convert_content_to_blocks` 再 `block_create_block_descendant` 追加。完成后强制进行标题、长段落锚点、图片数和公式表示四项线上对账。

## 🔗 组合契约（与 fetch-archive-to-lexiang，可选）

本 skill 的产物是**可移植 markdown 包**：`<title>.md`（含 `![](images/..)`、HTML 表、`> [!stat]` / `> [!definition]` 标注）+ `images/`。

归档 skill 消费时按下表映射成目标平台专有块，并负责「markdown ↔ 线上块」对账（凑成完整三方对账）：

| 本 skill 标注 | 归档渲染 |
|---|---|
| `> [!stat] **N**` | callout 块（📊） |
| `> [!definition] **term**` | callout 块（📖） |
| HTML/markdown 表格 | 原生表格块 |
| `![](images/..)` | 三步上传 + image 块 |

> 边界原则：**识别/裁剪/翻译/可移植标注**属本 skill；**专有块渲染/上传/线上增量改/线上对账**属归档 skill。
